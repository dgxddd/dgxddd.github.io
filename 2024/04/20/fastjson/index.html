Fastjson

Fastjson 是阿里巴巴开发、基于 Java 语言编写的高性能 JSON 处理库，主要用于实现 JSON 字符串与 Java 对象之间的相互转换。它没有使用 Java 原生的序列化机制，而是自定义了一套独立的序列化 / 反序列化方案，凭借极高的解析和转换性能、简洁易用的 API，曾被广泛应用在 Java 项目中。

fastjson 1.2.25

checkAutoType 安全机制

fastjson 1.2.25 引入了 checkAutoType 安全机制，采用黑白名单校验类是否可加载。默认情况下 autoTypeSupport 关闭，基于白名单实现安全机制，而打开 autoTypeSupport 之后，是基于内置黑名单来实现安全的，fastjson 也提供了添加黑名单的接口。

我们在ParserConfig下可以找到新引入的一些属性

autoTypeSupport 是新增的一个标志位，denyList 是黑名单，acceptList 是白名单。

其中黑名单如下有20个

ParserConfig#checkAutoType(String，Class<?>)。

第一个if开启自动类型解析autoTypeSupport=true，优先匹配白名单前缀，命中则直接加载类放行；白名单未匹配再校验黑名单前缀，命中则直接抛出异常拦截类的反序列化

先从全局类缓存中获取目标类，获取不到再从内置反序列化器缓存中查找，只要任意一处找到类就直接返回，完全跳过后续所有黑名单校验，仅在没找到类时才继续执行后面的安全检查。

在关闭 autoTypeSupport（默认状态）时强制执行的安全校验，会先遍历黑名单，只要类名前缀匹配就直接抛出异常拦截，拦截后不再执行任何后续逻辑；如果没命中黑名单，再去检查白名单，白名单匹配才允许加载类，不匹配则继续往下走。

如果上面的检查都通过了但是没有加载类的话，再检查一次 autoTypeSupport 是否为 true ，如果是就加载类。这是这个方法中最后一次加载类的地方了，也就是说如果 autoTypeSupport 为 false 的话，是不会加载任何类的。而 autoTypeSupport 默认为 false ：

首先对已加载的类做高危校验，直接禁止ClassLoader和DataSource及其子类反序列化；若存在预期类型则校验继承关系，符合则返回类；在默认关闭autoTypeSupport的场景下，无论前面逻辑如何，最终都会直接抛出异常拦截反序列化

我们查看就是TypeUtils#loadClass方法。第一种是数组格式类名（以[开头），会截取后续内容加载组件类型并生成数组类；第二种是Lxxx;包裹的类名，会自动去掉首尾的L和;，加载中间的真实类名，是 Fastjson 兼容特殊类名格式的核心处理代码。

因此就在这个位置出现了逻辑漏洞，攻击者可以使用带有描述符的类绕过黑名单的限制，而在类加载过程中，描述符还会被处理掉。因此，漏洞利用的思路就出来了：需要开启 autoType，使用以上字符来进行黑名单的绕过。
